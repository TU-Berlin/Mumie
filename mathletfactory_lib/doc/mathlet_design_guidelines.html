1. Allgemein

  • die Wahl eines Applet-Skeleton-Typs für ein Mathlet sollte eine wohlüberlegte Tat sein, da sich ein späteres Umstellen auf einen anderen Mathlet-Typen meist als sehr umfangreich darstellt.

  • alle Methoden eines Mathlets sollten mit dem private Attribut ausgestattet werden, um einen Missbrauch von Methoden-Aufrufen zu gewährleisten. Dies schließt nicht jene Methoden ein, die durch die Methoden-Vererbung bedingt nicht in ihrer Sichtbarkeit eingeschränkt werden können (wie z.B. die public-Methoden init() und reset()).

  • der "Einstiegspunkt" oder die erste aufgerufene Methode eines Applets ist die init()-Methode. Sie wird zum Initialisieren des Applets vom Web-Browser aufgerufen. Auszuführender Code muss also in dieser Methode geschrieben werden. Die vollständige Beschreibung lautet: 
    public void init()

  • um eine Applet-Skeleton-Klasse erfolgreich zu instantieren ist der Aufruf der init()-Methode der Super-Klasse des Mathlets (z.B. SingleG2DCanvasApplet, NoCanvasApplet) in der gleichnamigen Methode des Mathlets (am besten an erster Stelle) zu schreiben. D.h. in der init()-Methode eines Mathlets steht folgender Aufruf an 1. Stelle. 
    super.init();

  • Objekt-Definitionen sollten in der Methode 
    private void initializeObjects()
    stehen. Unter Umständen kann diese Methode auch in der reset()-Methode zum Neubesetzen von Werten dienen.

  • nach dem Initialisieren der (MM)-Objekte werden in der Methode 
    private void defineDependencies()
    die Dependecy-Adapter der MM-Objekte gesetzt.

  • die Zahlenklasse eines Mathlets sollte im Feld 
    private Class numberClass
    als 1. Variablen-Definition im Mathlet stehen. Durch Zuweisen von Werten wie z.B. MInteger.class oder MRational.class oder MComplex.class wird die Handhabe von MM-Objekten vereinfacht, da (fast) jedes von ihnen eine Klasse als Argument im Konstruktor beinhaltet. Durch das Benutzen einer einheitlichen Zalenklasse wird des weiteren die Flexibilität des Mathlets erhöht.

  • GUI-Komponenten wie z.B. Klick-Buttons, Panels, Labels, Renderer, ... sollten in einem Mathlet erst nach dem Aufruf von super.init() mit dem new Operator erstellt werden. Dies heißt insbesondere, dass Initialisierungen nicht im Kopf des Mathlets stattfinden dürfen.

2. Hausaufgaben

  • Hochpoppen des Applet-Fensters
    • beim Start des Applets
    • nach Interaktion mit Website ("select"-Methoden)

  • Bestätigung durch den Benutzer
    • wenn nicht alles ausgefüllt wurde und gespeichert werden soll
    • Löschen der Ergenisse eines Subtasks
    • Löschen aller Ergebnisse durch den Reset-Button

2. Rumpf für Corrector-Klassen

/**
 * @author Autor
 * 
 * @mm.category corrector
 * @mm.status devel_ok
 * @mm.description Deutsche Beschreibung
 * @mm.changelog 2005-10-05: Initial
 */

import net.mumie.mathletfactory.mmobject.number.MMInteger;
import net.mumie.mathletfactory.display.noc.number.MMNumberPanel;
import net.mumie.mathletfactory.util.BasicApplicationFrame;

import net.mumie.mathletfactory.appletskeleton.<Applet-Skeleton-Klasse>;

public class <Mathlet Name> extends <Applet-Skeleton-Klasse> {

  MMInteger number;
  MMNumberPanel numberPanel;

  public void init() {
    super.init();
    initializeObjects();

    addControl( numberPanel );
    ...
  }

  private void initializeObjects() {
    number = new MMInteger(2);
    number.setEditable(true);
    numberPanel = (MMNumberPanel) number.getAsContainerContent();
    numberPanel.setTextVisibleBeforeEdited(false);

    ...
  }

  public void reset() {
    number.setInteger(2);
    number.render();
    number.invokeUpdaters();

    ...
  }

  public static void main(String[] args) {
    <Mathlet Name> applet = new <Mathlet Name>();
    applet.init();
    BasicApplicationFrame frame = new BasicApplicationFrame(applet, <Fenster-Breite>, <Fenster-Höhe>);
    frame.show();
  }
}
